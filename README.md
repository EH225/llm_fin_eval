# LLM Agent Performance Evaluation on Financial Analysis Tasks
This repository contains code for a comparative performance analysis of LLM agents on a small set of financial analysis tasks. Below is a quick summary of the repo layout:
- `tasks`: This folder contains code that outlines the tasks given to each agent in this study and also outlines the expected answers. 
- `agents`: This folder contains code used to call the OpenAI and Anthropic APIs to process queries and run experiments.
-`runner.py` - This module is the main driver of the project and is where the experiments are run from.

- `analyze.py`: This module creates a performance summary JSON from the results cached to results and saved the new summary there as well. 


- `grader`: This folder contains code for performing automated evaluation of agent performance.
- `reports`: This folder contains HTML summaries of the study results.
- `results`: This folder contains cached JSON files detailing the results from each trial of each task.
- `dashboard.py`: This module summaries the information contained in the summary output from `analyze.py` into a readable HTML document and saves it to the `reports` folder.
- `environment.yml` - This file outlines the requirements of the conda env used to run the experiments of this project.

## Abstract

This project examines the head-to-head performance of LLM agents from OpenAI (gpt-5.4) and Anthropic (claude-opus-4-8) on a small suite of realistic financial-analysis tasks of varying difficulty utilizing public filings data. Each agent was asked to complete 6 tasks (2 easy, 2 medium, and 2 hard) for 10 trials per task. The tasks require general knowledge of financial analysis and are broadly indicative of the day-to-day tasks most junior a junior financial analysts would encounter. Where applicable, relevant data from public filings was provided as context within the input prompt. Rubric-based automated graders were used to evaluate the performance of the agents and failure analysis was conducted to understand where and how agents failed to return a satisfactory answer for certain tasks. Overall, the LLM agents exhibited excellent performance and were generally able to complete the task analyzed in this project, although with some variance and occasional failures.

## 1. Introduction

The introduction of the transformer architecture in 2017 by researchers at Google revolutionized deep learning-based approaches to natural language processing (NLP) by allowing models to process text in parallel and model long-range dependencies. While attention mechanisms had already been incorporated into RNN- and LSTM-based sequence models, these models were slow to train because their recurrent structure prevented efficient parallelization during training and they tended to struggle to capture relationships between distant tokens. The substantial improvement in performance offered by the transformer model design gave rise to a new generation of powerful large language models (LLMs) that out-performed all previous sequence-to-sequence architectures and ushered in a new age of AI capabilities. OpenAI's release of ChatGPT in late 2022 brought these advances to a mainstream audience, demonstrating the practical capabilities of large language models and accelerating investment and research across the field.

Since then, the capabilities of LLMs have steadily grown as model size, training data, and computational resources have steadily increased. Today, LLM-based agents are capable of performing a wide variety of tasks (e.g. translation, summarization, question answering, grammatical review, creative writing etc.) with high accuracy and can produce coherent multi-step solutions to complex problems. These models are actively being adapted to various domains and represent an important milestone toward developing more general-purpose artificial intelligence i.e. a system capable of learning and reasoning across a wide variety of domains at a level comparable to or exceeding that of humans, without task-specific retraining.

## 2. Agents

For the purposes of this project, I considered 2 LLM-based AI agents: OpenAI’s gpt-5.4 and Anthropic’s claude-opus-4-8. OpenAI and Anthropic are on the cutting edge of LLM development and have among the most performant and well-rounded agents currently available for consumer use. As of the time of this project, these agents are among the best currently offered. I accessed both through their python APIs to conduct my evaluation of their performance on a set of illistrative financial analysis tasks.

## 3. Tasks

Given my background in finance and economics, I was curious to explore and compare how well LLM agents would perform on a variety of financial-analysis tasks of varying difficulty utilizing public filings data. These tasks require general knowledge of financial analysis and are broadly indicative of the day-to-day tasks most junior a junior financial analysts would encounter. Where applicable, relevant data from public filings was provided as context within the input prompt.

1. Task 1: EPS Extraction (Tier 1 - Easy) - Given an excerpt from Apple's income statement, the agent is asked to report the diluted earnings per share (EPS), which is already stated in the income statement and can also be computed from the information provided.
2. Task 2: Revenue CAGR (Tier 1 - Easy) - Using Microsoft's annual revenue figures for fiscal years 2021, 2022, and 2023, the agent is asked to compute the 2-year compound annual growth rate (CAGR) of total revenue. There is a direct, closed-form solution so this task is straight-forward.
3. Task 3: Debt-to-Equity Ratio (Tier 2 - Medium) - Using Meta Platforms' balance sheet, the agent is asked to calculate the company's debt-to-equity ratio. This task is a bit more difficult since there is some challenge as to what should be included in the numerator and this financial ratio is not already reported in the filings data provided.
4. Task 4: Bond duration calculation (Tier 2 - Medium) - Given a bond's price, payout structure, and term, the agent is asked to compute its modified duration. This is difficult because it requires computing yield-to-maturity for which there is no general, closed-form solution and it requires the application of multiple formulas.
5. Task 5: Free Cash Flow Quality Analysis (Tier 3 - Hard) - Based on Alphabet's cash flow statement, the agent is asked to write a 150-250 word analysis of the quality of its FCF with some guidance of what to highlight. This tests the agent’s ability to extract out what is most relevant for the areas of analysis specified.
6. Task 6: Gross Margin Comparison (Tier 3 - Hard) - Using income statements for Apple, Microsoft, Alphabet, and Meta, the agent is asked to rank the companies by gross margin percentage (highest to lowest) and provide a 100-150 word explanation of what drives the differences. Here some general knowledge of the companies is required for the open-ended explanation component.

For each task, the agents were given access to a mathematical computation tool capable of evaluating expressions exactly (e.g. compute((5 + 4) / 2) returns 4.5). This serves several purposes: (1) it evaluates the agents' ability to invoke external tools appropriately, (2) it encourages them to decompose problems into explicit computational steps, making intermediate calculations visible, and (3) it improves numerical accuracy by performing arithmetic exactly rather than relying on the language model's approximate internal arithmetic.

For the written tasks, claude-sonnet-4-6 was used to evaluate the written responses and grade them programmatically using the criteria outlined in each task’s rubric. Normalized assessment scores were assigned, which ranged from 0.0 to 1.0.

## 4. Performance Evaluation

Overall, the LLM agents exhibited excellent performance on the financial analysis tasks and were generally able to consistently complete them with high accuracy. However, there was some variance in performance and occasional failures, according to the grader rubric. See the dashboard summary saved in `reports` for an interactive HTML document summarizing the performance of each agent on each task. Each task was run for 10 trials with each agent.

Both models performed quite well across a variety of tasks requiring both computational and written skills, it’s hard to say conclusively that one outperformed the other. Both agents appear to have used around the same number of turns per task, but OpenAI’s agent tended to have a much lower runtime (latency) compared to the Claude agent, often being 2x or 3x faster on average to return a final answer. Along that dimension, the OpenAI GPT agent was a clear winner.

## 5. Failure Analysis
Examining how and why agents fail to produce satisfactory responses is essential for understanding both their capabilities and their limitations. Analyzing failure cases provides valuable insight into the circumstances under which an agent's reasoning, planning, tool usage, or decision-making breaks down. These analyses can reveal systematic weaknesses, identify opportunities for improving model architectures and prompting strategies, and guide the development of more robust and reliable agentic systems. Consequently, evaluating failure modes is as important as measuring overall task performance when assessing the effectiveness of AI agents.

In this study, the Claude Opus model achieved 100% accuracy on all tasks except the Debt-to-Equity Ratio calculation. Closer examination of the execution logs revealed that the agent returned a value of 0.44 instead of the expected 0.055 in 3 of the 10 trials. This occurred because the agent used the Total Liabilities line item from the balance sheet rather than only Long-Term Debt in the numerator. While the expected answer was based on the latter definition, the former is also commonly used in finance and accounting, making the task inherently ambiguous. Had the prompt explicitly specified which definition of debt-to-equity to apply, the agent would likely have returned the expected value consistently. This result highlights an important consideration when evaluating AI agents: performance can be influenced not only by the model's reasoning ability, but also by ambiguities in the task specification. In domains where multiple valid methodologies exist, clear and precise instructions are essential for obtaining consistent and reproducible results.

The failure cases observed for the OpenAI GPT model differed from those of Claude Opus and were primarily associated with the bond duration calculation and free cash flow (FCF) quality analysis tasks. The bond duration task was computationally more demanding than the other financial calculations because determining a bond's yield-to-maturity (YTM) requires solving a nonlinear equation for which no general closed-form solution exists. Consequently, both agents adopted an iterative guess-and-check strategy, repeatedly estimating the YTM and using the bond pricing equation to compare the resulting price against the value provided in the prompt. In all three unsuccessful trials, the OpenAI GPT agent reached the maximum turn limit of 10 before returning a final answer. Examination of the execution logs indicated that the agent was converging toward the correct solution, but was unable to complete the iterative process before being truncated. These failures therefore appear to reflect a limitation of the execution environment rather than the model's underlying reasoning capability. They could likely be mitigated by increasing the permitted number of turns, informing the agent of the turn limit in advance so it can adopt a more efficient strategy, or providing access to a dedicated numerical root-finding tool.

The OpenAI GPT model also performed well on the FCF quality analysis task, with only one trial failing to meet the required performance threshold. In that instance, the response received a score of 83%, narrowly missing the required 85%, primarily because it slightly exceeded the specified word limit and did not sufficiently address several of the desired evaluation criteria. Overall, the model demonstrated strong performance on this task. However, because the responses were evaluated by a Claude-based agent, it is possible that the evaluation exhibited a modest preference for Claude's writing style or response characteristics, representing a potential source of systematic bias when interpreting these results.

## 6. Conclusion
Overall, both LLM agents demonstrated strong performance across the financial-analysis tasks examined in this project, achieving high success rates on the majority of trials, even those involving computationally intensive or multi-step reasoning tasks. While the OpenAI GPT agent generally ran faster, there was no overall clear winner from a performance perspective. Where failures did occur, it was generally on tasks where higher specificity in the prompt would have likely been a mitigating factor. While both models proved capable of performing many tasks representative of a junior financial analyst's workflow, occasional errors highlight the importance of human oversight, particularly for calculations and analyses involving multiple dependent reasoning steps. These results suggest that modern LLMs are valuable tools in this domain and have a substantial ability to complete standard tasks in financial analysis.
